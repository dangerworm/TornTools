## Summary

The session delivered a meaningful structural improvement: the single `QueueProcessor` was replaced
with `TornMarketsProcessor` + `Weav3rBazaarsProcessor` backed by a shared `QueueProcessorBase`, the
Python Weav3r caller was converted from per-call process spawning to a persistent sidecar, and
several long-standing correctness bugs (silent JSON parse failures in `dotnetapi.ts`, missing input
model validation, `RemoveQueueItemsAsync` deleting non-Failed items) were fixed. The core
concurrency model is sound. A few issues deserve attention before this gets used under load.

**`CreateQueueItemsAsync` interleaving:** formula `withinGroupIndex * typeCount + (int)callType`

### bb37c68a — Add parallel workers and persistent Python server

The two biggest changes in the session.

**`Weav3rPythonServer`:** Singleton managing a persistent `bazaar_server.py` subprocess. Protocol is
line-delimited JSON over stdin/stdout. Timeout is 30 s with a `CancellationTokenSource` linked to
the caller token. On timeout, the process is killed and restarted on next call. `BeginErrorReadLine`
wires stderr to the logger. `IDisposable` is correctly implemented with `KillProcess()` on dispose.
Serialisation via `SemaphoreSlim(1, 1)` is correct.

**`QueueProcessor`** refactored into `QueueProcessorBase` + two concrete processors. The repopulate
lock was correctly changed from `static` to instance-level (`_repopulateLock = new(1, 1)`) — the old
`static` lock would have caused the two processors to contend with each other even though they own
separate call-type partitions.

### 2498f773 — Fix repopulation data loss and rate-limit scaling

Two correctness fixes that should have been in the previous commit:

1. **`HasInProgressItems(callType)` guard:** without this, a processor that wins the repopulate lock
   while sibling workers hold InProgress rows would call `RemoveQueueItemsAsync(callType)` and then
   repopulate — correctly scoped now since it only touches its own call type. But a subtle point: at
   the time of this fix, `RemoveQueueItemsAsync(callType)` only removes **Pending** rows, not
   InProgress ones. So even without the guard, in-flight work would not be deleted. The guard is
   still the right defence — it avoids unnecessary repopulation that would reset ordering.

2. **Rate-limit delay scaling:** `CalculateDelayMilliseconds` now takes `workerCount` and multiplies
   the per-worker sleep by it. Correct: each of N workers sleeps N× longer so combined throughput
   stays within budget.

### a7187d9e — Split QueueProcessor into TornMarketsProcessor and Weav3rBazaarsProcessor

Final architectural shape. The `GetNextQueueItemAsync` SQL gains `AND call_type = @callType`,
`HasInProgressItemsAsync` and `RemoveQueueItemsAsync` are scoped by call type, and
`PopulateQueueWithMarketAndWeav3rItemsOfInterest` is split into `PopulateQueueWithTornMarketItems` /
`PopulateQueueWithWeav3rItems`.

`infra/app_service.tf` adds `TornMarketsProcessorConfiguration__WorkerCount` and
`Weav3rBazaarsProcessorConfiguration__WorkerCount` env vars — lets operations scale without
redeploy.

## Issues and concerns

### 1. `ApiCallType.Ignore` is still in the enum but `typeCount` includes it

`Enum.GetValues<ApiCallType>().Length` counts `Ignore`, `TornItems`, and `YataForeignStock` even
though none of those have processors or are inserted via `CreateQueueItemsAsync`. This makes the
index formula's comment ("without any change to this method when new `ApiCallType` values are
added") misleading — adding `TornItems` as a processor would assign it index offset 1, colliding
with no current type, but the comment implies adding future types is safe regardless of where in the
enum they land. The actual ordering invariant is simply "indices within a call type are ascending",
which holds. Consider replacing the formula comment with that simpler statement.

### 2. `PopulateQueueWithTornMarketItems` falls back to stale scan; `PopulateQueueWithWeav3rItems` does not

TML falls back to `PopulateQueueWithStaleMarketItems` when no active items are found. WBL silently
returns without populating. If Weav3r data goes stale (no change logs in 7 days), the WBL processor
will spin in a tight repopulate → no items → repopulate → … loop, each iteration burning 4 database
queries and a 500ms delay. Not catastrophic but noisy. A log-and-wait, or a WBL-specific stale scan
fallback, would be cleaner.

### 3. `Weav3rPythonServer` with `WorkerCount > 1` is effectively serialised

`_requestLock = new(1, 1)` means only one Weav3r HTTP call runs at a time. If
`Weav3rBazaarsProcessorConfiguration.WorkerCount` is bumped above 1, additional workers queue behind
the lock, adding latency without throughput gain (Weav3r's `MaxCallsPerMinute` is also a cap, so
more workers would not increase call rate anyway). The delay formula already accounts for
`workerCount`, so throughput stays correct, but extra workers for WBL are wasted. This is low risk
at the current default of 1 but worth a comment near `WorkerCount` in
`WeaveRBazaarsProcessorConfiguration` or in `WeaveRBazaarsProcessor`.

### 4. `GetDelayMillisecondsAsync` is called with a potentially disposed scope

In `QueueProcessorBase.RunWorkerAsync`, `GetDelayMillisecondsAsync(databaseService, stoppingToken)`
is called after the outer `try/catch` but still inside `using var scope = ...`. The scope is
`IDisposable` and the `using` goes until the end of the `while` block, which is _after_ the delay
call. So the `databaseService` is valid here. No bug, but the code reads as if the delay is outside
the scope's lifetime. Adding a comment would help.

### 5. `DEFAULT_MAX_TIME_INDEX = 2` (5 minutes) is a very tight default

Since localStorage persistence was also introduced in the same session, **all users** effectively
have no stored preference and will get the 5-minute default on their next visit. The Resale page
will look empty until items have been scanned within the last 5 minutes, which may not be true
immediately after a deployment or at low-activity periods. Consider defaulting to 10–15 minutes
(index 3 or 4) or adding a "no matching items" placeholder that explains the filter.

### 6. `loadSliderIndex` is called on every render

```tsx
const initialMinProfitIndex = loadSliderIndex(...)
const [minProfit, setMinProfit] = useState(PRICE_RANGE_VALUES[initialMinProfitIndex])
```

`loadSliderIndex` calls `localStorage.getItem` on every render, but `useState`'s initial value is
only used on the first render. Idiomatic React uses a lazy initialiser:

```tsx
const [minProfit, setMinProfit] = useState(() => {
  const idx = loadSliderIndex("resale:minProfit:v1", DEFAULT_MIN_PROFIT_INDEX, PRICE_RANGE_VALUES);
  return PRICE_RANGE_VALUES[idx];
});
```

Not a correctness bug (localStorage access is synchronous and fast), but worth fixing for clarity.

### 7. `RemoveQueueItemsAsync()` (no callType arg) is called at startup for both types

`Program.cs:72` calls `databaseService.RemoveQueueItemsAsync(CancellationToken.None)` on startup,
which removes all Pending queue items regardless of call type. This is correct for a clean-slate
start after a crash. Confirm that any InProgress items left from a previous crash are handled
gracefully — currently they would remain InProgress indefinitely since no processor would clean them
up (both processors only dequeue Pending rows). A startup job or a `HasInProgressItems` →
`MarkAbandoned` pass would address orphaned InProgress rows.

### 8. No test project

Noted in the session handoff, but the concurrency changes introduced in this session (SELECT FOR
UPDATE SKIP LOCKED, repopulate guard, rate-limit scaling) are exactly the kind of logic that
benefits most from integration tests against a real PostgreSQL instance. The new abstract base class
structure makes unit-testing the worker loop straightforward via a mock `IDatabaseService`.

## What was done well

- **`SELECT FOR UPDATE SKIP LOCKED`** is the right primitive. The raw SQL via `NpgsqlCommand` is
  correctly scoped (connection opened and closed in a finally block), parameterised, and returns a
  fully-mapped DTO without a second round-trip.

- **Abstract base processor** is clean. `CallType`, `WorkerCount`, `RepopulateAsync`, and
  `GetDelayMillisecondsAsync` are all well-chosen seams. Adding a third call type requires only a
  new 30-line concrete class and a config type.

- **`HasInProgressItems` guard** is a good defensive layer even if it turns out to be redundant due
  to the type-filtered `RemoveQueueItemsAsync`.

- **`Weav3rPythonServer` error handling:** crash → kill → restart-on-next-call is simple and
  correct. The 30 s timeout with a `CancellationTokenSource.CreateLinkedTokenSource` is idiomatic.

- **`dotnetapi.ts` fix** is a clear improvement with no ambiguity.

- **Input model validation** is comprehensive and covers the likely attack surface (API key length,
  hex colours, enum modes).

- **Session handoff document** is excellent. The "outstanding from previous sessions" section in
  particular gives the next session a clean, actionable starting point.
